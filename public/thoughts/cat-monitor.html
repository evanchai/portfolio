为什么用边缘 AI

Why Edge AI

家里两只猫——萨摩和小白——老是偷偷上沙发。需要实时监控 + 即时告警，但传统方案有明显短板：延迟高、依赖网络、隐私问题。

Two cats at home — Samo and Xiaobai — keep sneaking onto the sofa. Real-time monitoring plus instant alerts are needed, but the traditional approach has obvious drawbacks: high latency, network dependency, and privacy concerns.

边缘方案的核心：IMX500 芯片直接在传感器上跑神经网络，推理延迟 <100ms，不需要上传原始图像到任何云端。

The core of the edge approach: the IMX500 chip runs neural networks directly on the sensor, with inference latency <100ms, no need to upload raw images to any cloud.

传统方案Traditional Camera → Upload → Cloud GPU → Result ↓ latency: 1-5s, network dependent, privacy risk 边缘方案Edge approach Camera + IMX500 → Local Result → Only alerts go to cloud ↓ latency: <100ms, offline capable, images stay local

两级检测架构

Two-Stage Detection

单靠边缘芯片的检测精度有限，单靠云端 AI 又太慢太贵。解决方案：两级级联，各司其职。

Edge chip detection alone lacks precision. Cloud AI alone is too slow and expensive. The solution: a two-stage cascade, each doing what it does best.

第一级：IMX500 上的 EfficientDet Lite0

Stage 1: EfficientDet Lite0 on IMX500

  • 在芯片上运行，零延迟，低功耗（<5W 整机）
  • 输出：bounding boxes, confidence scores, class IDs（COCO 80 类，cat=15, dog=16）
  • 阈值设为 0.3——偏低，宁可误报不漏报
  • 同时检测猫和狗，因为边缘模型无法区分具体是哪只
  • Runs on-chip, zero latency, low power (<5W total system)
  • Outputs: bounding boxes, confidence scores, class IDs (COCO 80 classes, cat=15, dog=16)
  • Threshold set to 0.3 — intentionally low, prefer false positives over missed detections
  • Detects both cats and dogs, since the edge model can't distinguish which specific pet

第二级：Gemini 2.5 Flash 视觉分析

Stage 2: Gemini 2.5 Flash Visual Analysis

  • 只在第一级触发时调用（节省 API 调用）
  • 输入：12MP 高清照片
  • 输出：是否在沙发上 + 哪只猫 + 在做什么
  • 5 分钟冷却期，避免重复调用
  • Only called when Stage 1 triggers (saves API calls)
  • Input: 12MP high-resolution photo
  • Output: on sofa or not + which cat + what it's doing
  • 5-minute cooldown to avoid redundant calls
// Gemini prompt format SYSTEM: You are analyzing a photo from a pet monitoring camera. Determine if a cat or dog is on the sofa/couch. Known pets: - "Samo": orange tabby cat - "Xiaobai": white cat Respond in this exact format: YES|cat_name|description of what the pet is doing NO|none|description of scene // Example output: YES|Samo|Orange tabby curled up sleeping on the left cushion
为什么不直接用 Gemini 做检测？Why not use Gemini directly for detection? Gemini 需要 1-3s 响应，而且有 API 限额（免费 10 RPM）。IMX500 可以每秒扫描 2 次，零成本。边缘粗筛 + 云端精分 = 最优解。 Gemini takes 1-3s to respond and has API rate limits (free tier: 10 RPM). The IMX500 scans twice per second at zero cost. Edge coarse filter + cloud fine classification = optimal solution.

实时数据流

Real-Time Data Pipeline

从树莓派到浏览器，数据流经三个节点。Redis 是核心中继。

From Raspberry Pi to browser, data flows through three nodes. Redis is the core relay.

完整数据流Full pipeline Pi (monitor.py) → Redis (3 keys) → Vercel API → React Dashboard Redis key design: cat:heartbeat SET, TTL 120s → status, scan count, detection count, logs cat:snapshot SET, TTL 10s → base64 image, updated every second cat:events LIST, max 20 items → detection events + pipeline steps
为什么用 Redis TTL？Why Redis TTL? 如果 Pi 断电，cat:snapshot 10 秒后自动过期，Dashboard 自动显示 "Offline"。cat:heartbeat 120 秒后过期。无需额外的离线检测逻辑——TTL 本身就是心跳机制。 If the Pi loses power, cat:snapshot auto-expires after 10 seconds and the Dashboard shows "Offline". cat:heartbeat expires after 120 seconds. No separate offline detection logic needed — TTL itself is the heartbeat mechanism.

Upstash Redis REST API 的两种用法

Two Ways to Use Upstash Redis REST API

Pi 端写入大 payload 用 POST body，Vercel 端读取用 GET path：

Pi-side writes large payloads via POST body; Vercel-side reads via GET path:

// Pi side — writing large payloads (base64 images) POST https://redis-url/ Body: ["SET", "cat:snapshot", "<base64_data>", "EX", "10"] // Vercel side — reading events GET https://redis-url/LRANGE/cat:events/0/19 // Vercel side — reading heartbeat GET https://redis-url/GET/cat:heartbeat
踩坑：Pitfall: POST 方式的 LRANGE 对大 payload 返回空结果，GET path 方式正常。怀疑是 Upstash REST 序列化的 bug。最终规则：写用 POST body，读用 GET path。 POST-based LRANGE returns empty results for large payloads, while GET path works fine. Suspected Upstash REST serialization bug. Final rule: write with POST body, read with GET path.

模型升级：nanodet → EfficientDet

Model Upgrade: nanodet → EfficientDet

最初使用 nanodet，cat class 的置信度只有 0.15-0.27，最高也才 0.35。阈值从 0.4 降到 0.3 勉强能用，但误报也增加了。

Initially used nanodet, but cat class confidence was only 0.15-0.27, maxing out at 0.35. Lowering the threshold from 0.4 to 0.3 barely worked, but false positives also increased.

IMX500 上可用的 _pp（post-processed）检测模型对比：

Available _pp (post-processed) detection models on IMX500:

ModelCOCO mAPNotes

efficientdet_lite0~26最终选择，精度最高Final choice, highest accuracy
ssd_mobilenetv2_fpnlite_320x320~22中规中矩Middle ground

切换只需改一行模型路径，输出格式完全兼容（boxes/scores/classes）。EfficientDet 的 cat 置信度稳定在 0.4-0.7，远好于 nanodet。

Switching requires changing just one model path line — output format is fully compatible (boxes/scores/classes). EfficientDet's cat confidence is stable at 0.4-0.7, far better than nanodet.

关于 _pp 后缀：About the _pp suffix: IMX500 的 _pp 后缀 = post-processed，模型输出已经是 boxes/scores/classes，不需要自己做 NMS（Non-Maximum Suppression）。直接拿来用，大幅简化代码。 IMX500's _pp suffix = post-processed. Model output is already boxes/scores/classes — no need to implement your own NMS (Non-Maximum Suppression). Just use it directly, massively simplifying the code.

结论：Verdict: 边缘推理做粗筛 + 云端 AI 做精分，是个人 IoT 项目的最优解。IMX500 的零成本检测解决了"要不要调 API"的犹豫——反正本地跑不花钱，检测到了再花钱问 Gemini。 Edge inference for coarse filtering + cloud AI for fine classification is the optimal solution for personal IoT projects. 总结

Conclusion

最优架构Optimal architecture Edge detection Cloud AI (free, fast, always-on) (smart, expensive, on-demand) \ / \ / → Best of both worlds ← | Real-time monitoring + Intelligent alerts

这个项目的核心洞察：不要在边缘和云端之间二选一，而是让它们各做最擅长的事。

The core insight of this project: don't choose between edge and cloud — let each do what it does best.

IMX500 以零成本、每秒两次的频率扫描画面。99% 的时间什么都没有，芯片静默运行。一旦检测到猫，才唤醒 Gemini 做精细判断——是哪只猫、在不在沙发上、在做什么。整个系统功耗不到 5W，一根 USB-C 线供电。

The IMX500 scans the frame at zero cost, twice per second. 99% of the time there's nothing, and the chip runs silently. Only when a cat is detected does it wake up Gemini for fine classification — which cat, whether it's on the sofa, what it's doing. The entire system draws less than 5W, powered by a single USB-C cable.

从硬件调参到全栈数据流，从图像压缩到模型选型——一个看似简单的"检测猫上沙发"需求，最终变成了一次从边缘到云端的完整工程实践。

From hardware tuning to full-stack data pipelines, from image compression to model selection — a seemingly simple "detect cat on sofa" requirement turned into a complete engineering journey from edge to cloud.
